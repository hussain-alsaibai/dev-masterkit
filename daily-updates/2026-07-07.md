### New repos (live on GitHub)

#### tiny-policy — `github.com/hussain-alsaibai/tiny-policy`
Zero-dependency **attribute-based authorization** (ABAC) policy engine
for AI agents and HTTP APIs. JSON policy files, glob matching, deny-overrides.
~190 LOC, no deps, ~1.5 µs per evaluation.

- 11 condition ops: `eq`, `ne`, `in`, `nin`, `gt`/`gte`/`lt`/`lte`,
  `contains`, `startsWith`, `endsWith`, `matches` (glob), `exists`
- Glob patterns: `*` (single segment, no dot), `**` (multi-segment), `?`
- **Deny-overrides** semantics — explicit deny wins over any allow
- Default-deny; opt-in `{ default: 'allow' }` for "open now, lockdown later"
- `evaluate()` / `explain()` / `metrics` (allow/deny counts, last decision)
- `PolicyLoader.load(dir)` reads all `*.json` files in a directory
- Static condition values — documented limitation; pair with a higher
  layer for dynamic cross-attribute references
- 19/19 tests passing; benchmark included

**Why now?** Every agent that touches real data eventually asks *"should it?"*
and the answer isn't an RBAC role list — it's a function of subject
attributes + resource attributes + action. OpenFGA / Cerbos / Cedar are
the heavy answers; `tiny-policy` is the fits-in-one-config-file answer.

#### tiny-otel — `github.com/hussain-alsaibai/tiny-otel`
Zero-dependency **OTLP/HTTP trace exporter** for AI agents. Ships the
exact `ExportTraceServiceRequest` envelope from the OTLP v1.5.0 spec —
every collector that speaks OTLP accepts it. ~268 LOC, no deps.

- `inSpan(name, asyncFn)` wrapper with auto status + exception capture
- `recordException(err)` adds OTLP-spec exception event + sets ERROR
- `setAttribute(k, v)` with typed AnyValue encoding (string/int/double/bool/array/kvlist)
- Standard `OTEL_EXPORTER_OTLP_ENDPOINT`, `OTEL_EXPORTER_OTLP_HEADERS`,
  `OTEL_SERVICE_NAME` env vars honored
- Batched export with configurable batch size + flush interval
- **Fetch injection** for testing — no global monkey-patching
- 10/10 tests passing including a real HTTP server integration test

**Why now?** The official `@opentelemetry/exporter-trace-otlp-http` is a
6 MB dep tree just to POST a JSON envelope. For a cron-style agent that
wants to ship a handful of spans to its own collector, that's overkill.
Honeycomb / Tempo / SigNoz / Datadog (via otel-collector) / Jaeger all
accept the envelope `tiny-otel` produces.

## Trend call

The "agent control plane" primitives the stdlib and big frameworks don't
ship well continue to converge on three tiers:

1. **Capability enforcement** — what an agent can spend (tiny-budget),
   what an agent can call (tiny-rate), how to retry safely (tiny-retry),
   **what an agent is allowed to do at all (tiny-policy)**
2. **Durable state** — durable pub/sub (tiny-eventbus), persistent queue
   (tiny-queue), crash-safe idempotency keys (tiny-idempotency),
   atomic cost persistence (tiny-budget)
3. **Observability** — structured logs (tiny-log), metrics (tiny-metrics),
   **traces shipped to a real collector (tiny-otel)**

Today's two additions close the obvious remaining gaps in tiers 1 and 3.
Tier 2 is well-covered.

### Where the fleet is still thin (next session priorities)

- **Schema registry** — agent tool calls benefit from a small JSON-Schema
  validator per tool. Could be a `tiny-schema` or an extension to
  `tiny-validator`.
- **W3C trace context propagation** across HTTP — `tiny-trace` (Python)
  has it; the Node side doesn't yet.
- **Cross-attribute policy conditions** — `tiny-policy` condition values
  are static; a tiny `$(resource.tenant)` template layer would let one
  rule express "subject's tenant equals the resource's tenant".
